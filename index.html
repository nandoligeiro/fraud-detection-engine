Demonstração técnica do MVP

Motor antifraude em tempo real, explicável e orientado a eventos.

Uma arquitetura para receber transações rapidamente, desacoplar a decisão por Kafka e gerar alertas com regras determinísticas auditáveis.

8k TPS médio esperado
25k TPS em pico
500ms janela-alvo de alerta
01 Problema

Antes da tecnologia, a jornada é de risco.

Cada transação carrega sinais de comportamento, valor, canal e localização. O motor precisa decidir se há suspeita sem prender o canal de entrada.

Transaction

Representa identificadores, valor, moeda, merchant, canal, localização e horário.

transactionId accountId cardId

Rule

Define critérios determinísticos, prioridade, versão, severidade e status.

HIGH_AMOUNT CARD_NOT_PRESENT

Fraud

Consolida decisão, alerta, severidade e regras acionadas para investigação.

SUSPICIOUS FraudAlert
Receber rápido, desacoplar por evento e decidir de forma explicável.
02 Arquitetura

Hexagonal por responsabilidade, event-driven por fluxo.

O desenho separa decisão de negócio de detalhes técnicos. O domínio não conhece Spring, Kafka, Redis ou banco.

Domain

Transação, regra, motor determinístico, decisão, severidade e alerta.

Application

Casos de uso, comandos e portas para orquestrar ingestão e detecção.

Infrastructure

Adapters REST, Kafka, logging, memória e configuração Spring.

Boundaries

Kafka marca a fronteira entre receber a transação e decidir a fraude.

│   ├── transaction
│   └── detection
├── domain
│   ├── transaction
│   ├── rule

└── infrastructure

    └── config
03 Ingestão

A API aceita a transação, controla duplicidade e publica o evento.

O request HTTP não executa a decisão antifraude completa. Ele transforma o payload em comando, aplica o guard e entrega a transação para processamento assíncrono.

1ClientEnvia a transação financeira.
2REST AdapterValida payload e trace.
3Use CaseCria o comando de ingestão.
4GuardEvita efeito duplicado.
5KafkaPublica em transaction-events.
Eu não tentei vender exatamente-once. Preferi assumir at-least-once delivery e garantir efeito effectively-once com idempotência.
04 Kafka

Kafka é a fronteira entre ingestão e decisão.

O broker absorve variação de carga, permite escalar consumers e evita que lentidão de regra, cache, banco ou notificação contamine o canal de entrada.

O que melhora

Desacoplamento, replay, particionamento por chave, escala horizontal e isolamento operacional.

O que cobra

Idempotência, consumer lag, DLQ, retry policy e observabilidade real do fluxo.

05 Detecção

Regras determinísticas primeiro: explicáveis, testáveis e auditáveis.

O consumer reconstrói o domínio, chama o caso de uso de detecção, carrega regras por porta e deixa o motor de domínio avaliar a transação.

1transaction-eventsEvento aceito para avaliação.
2Kafka ConsumerMapeia payload para domínio.
3RuleProviderPortCarrega regras ativas.
4Rule EngineAvalia critérios determinísticos.
5FraudAlertFactoryCria alerta quando há suspeita.
DefaultRuleProviderAdapter DeterministicRuleEngine FraudDetectionService FraudAlertFactory
06 Alerta

O alerta vira evento e chega a um consumidor independente.

O MVP não implementa push, e-mail ou backoffice reais. Ele prova que a decisão antifraude gera um alerta publicável para canais consumidores.

Payload

alertId, transactionId, accountId, severidade, decisão, regras acionadas e timestamps.

Tópico

fraud-alerts representa o contrato de saída para canais internos e externos.

Simulador

KafkaFraudAlertConsumer confirma que o alerta chega ao próximo componente.

07 MVP

O caminho feliz está demonstrável de ponta a ponta.

A versão atual comprova a arquitetura central sem fingir que toda a complexidade produtiva já foi implementada. 

REST /v1/transactions
  → TransactionIngestionService
  → Kafka transaction-events
  → KafkaTransactionEventConsumer
  → FraudDetectionService
  → DefaultRuleProviderAdapter
  → DeterministicRuleEngine
  → FraudAlertFactory
  → Kafka fraud-alerts
  → KafkaFraudAlertConsumer

Validado

Recebimento REST, publicação Kafka, consumo assíncrono, regras e alerta.

Flags

Kafka, publisher, consumer de alerta e regras default são habilitados por configuração.

Comando

make up, make kafka-topics e make run-mvp formam a jornada local.

08 Evolução

O MVP deixa os próximos adapters claramente posicionados.

As lacunas produtivas não mudam o domínio: entram como adapters, políticas operacionais e contratos de infraestrutura.

Redis

Idempotência distribuída com SET NX EX, suportando múltiplos pods e retries.

PostgreSQL

Persistência de regras, auditoria de alertas, versões, severidade e decisão tomada.

Rule Admin API

Criação, alteração, versionamento e kill switch de regras sem redeploy.

DLQ e retry

DefaultErrorHandler, DeadLetterPublishingRecoverer e reprocessamento seguro.

Observabilidade

OpenTelemetry, consumer lag, latência de detecção, p95/p99 e métricas de negócio.

Testcontainers

Testes de integração com Kafka, Redis e PostgreSQL para validar infraestrutura real.

09 Trade-offs

As escolhas foram simples de demonstrar e honestas para evoluir.

Cada decisão reduz alguma complexidade inicial, mas preserva uma rota clara para produção.

Síncrono vs assíncrono

Assíncrono ganha desacoplamento e escala, mas exige lag, DLQ, retry e rastreabilidade.

Regras vs ML

Regras determinísticas ganham explicabilidade e auditabilidade; ML pode entrar depois como score híbrido.

Memória vs Redis

Memória fecha o MVP; Redis torna a idempotência distribuída entre instâncias.

Default rules vs banco

Regras fixas provam o fluxo; PostgreSQL e Rule Admin API resolvem governança.

Resumo da demo

A solução é um motor antifraude event-driven. A API recebe transações, publica em Kafka, um consumer executa regras determinísticas no domínio e, quando há suspeita, publica um alerta em outro tópico. Organizei com arquitetura hexagonal: domínio puro, aplicação com casos de uso e infraestrutura com REST e Kafka. O MVP fecha o fluxo ponta a ponta; as evoluções naturais são Redis para idempotência, PostgreSQL para regras e auditoria, DLQ para resiliência e OpenTelemetry para observabilidade.

O MVP é simples o suficiente para validar a solução, mas estruturado o suficiente para crescer sem acoplar API, regra, Kafka e banco.
